# AGENTS.md

この文書は、このリポジトリを扱う Codex およびその他の開発エージェント向けの作業規約です。  
安全性、既存データとの互換性、iPhone Safari での実用性を優先してください。

## 1. プロジェクト概要

このリポジトリには、独立した2つのローカルファーストWebアプリがあります。

- `runos/RunOS v100 apex.html`
  - ランニング履歴、体調、目標レースから走力・疲労・練習計画を分析するアプリ
- `wanoku-navi/wanoku navi v27 yakou.html`
  - 東京湾奥の潮汐、季節、天候、釣果、移動時間などから釣行判断を支援するアプリ

現状はどちらも、CSS、HTML、状態管理、計算ロジック、描画、イベント処理を単一HTMLに内包しています。ビルドシステム、型検査、自動テストはありません。

`wanoku-navi/SETUP 自動取得.md` は外部情報取得用Cloudflare Workerのセットアップ資料ですが、文書が参照するWorker本体はこのリポジトリに含まれていません。また、一部のファイル名は旧バージョンを参照しています。

## 2. RunOS の目的と注意点

RunOSは、ランニングのトレーニング意思決定を支援するアプリです。主な対象は以下です。

- ランニング活動、ウェルネス、レース結果の記録
- FITファイル、Strava CSVの取込
- TRIMP、CTL、ATL、TSB、ACWRなどの負荷分析
- eVO2max、Critical Speed/Power、耐久性、効率などの走力分析
- 今日の練習提案、週間計画、シーズンプラン、レース予測
- GPS、ラップ、心拍、速度、パワー、インターバルの分析
- シューズ、天候、暑熱、補給、回復に関する支援

注意点:

- 全データは基本的に `meridian.v1` という単一の `localStorage` キーへ保存されます。
- FIT由来のストリーム、GPS、ラップ、曲線データも保存対象です。容量増加と保存失敗に注意してください。
- 現在の保存処理は例外を通知せず無視するため、「画面上は成功したが永続化されていない」状態が起こり得ます。
- 保存データ破損時のフォールバック、デモデータ生成、バックアップ読込は、既存ユーザーデータを失う可能性がある重要経路です。
- 分析式や判定閾値は互いに連動しています。単独の数値変更でもToday、Coach、Fitness、Season、Insightsへ波及します。
- FIT解析や活動データの正規化を変更する場合、既存活動との重複判定、単位、欠損値、旧データ互換性を必ず確認してください。
- Open-Meteoと位置情報を利用する機能があるため、完全な無通信アプリではありません。

## 3. wanoku-navi の目的と注意点

wanoku-naviは、東京湾奥を中心とした釣行判断支援アプリです。主な対象は以下です。

- 魚種、日時、潮汐、季節、水温、天候、地形、ベイトによる釣行スコア
- スポットランキング、短時間釣行、ランガン順序、撤退判断
- ピンスポット、広域傾向、狙い目時間、複数日予測
- Leafletと地図タイルを使ったスポット表示
- 実釣ログ、外部釣果、公式情報、予測検証
- 実績に基づく重み調整、ソース信頼度学習
- JMA、Anthropic API、Cloudflare Worker、RSS等からの外部情報取得

注意点:

- アプリ状態はグローバルな `S` に集約され、多数の描画関数と計算関数が直接参照します。
- `innerHTML` による再描画後、イベントハンドラを付け直す実装が多くあります。UI変更時は操作不能や二重登録に注意してください。
- 潮汐、スコア、信頼度、移動時間、外部シグナルは相互依存しています。スコア式の変更はランキング、Cockpit、ルート、予測検証へ波及します。
- 同名関数や類似ロジックが存在し得ます。追加前に既存定義を検索してください。
- Leaflet、Google Fonts、地理院・OSMタイルなどのCDN依存があり、オフライン時は表示が変わります。
- Cloudflare Worker本体は現リポジトリにありません。存在しないWorkerを前提に機能完了と判断しないでください。
- APIモデル名、JMA JSON、CORS、Workerのレスポンス形式など、外部仕様は変更され得ます。失敗時の表示とフォールバックを維持してください。

## 4. 変更時の原則

1. 変更前に、対象機能の入力、保存先、呼出元、描画先を検索して影響範囲を把握する。
2. 既存ユーザーデータとの後方互換性を最優先する。
3. 一度に扱う目的を1つに絞り、差分を小さく保つ。
4. 既存の計算結果を変える変更と、見た目だけの変更を同じ差分に混ぜない。
5. 現行のキー名、ID、データ形状、バックアップ形式を理由なく変更しない。
6. 保存形式を変更する場合は、バージョン、移行処理、失敗時の復旧方法を先に設計する。
7. 外部通信は失敗を通常ケースとして扱い、オフラインでも主要画面が壊れないようにする。
8. 例外を無言で握りつぶす箇所を増やさない。ユーザーデータに関わる失敗は確認可能にする。
9. DOMへ外部入力やインポート値を挿入する場合は、必ずエスケープまたは安全なDOM APIを使う。
10. コメントやドキュメントも、実装と同じ差分内で整合させる。
11. 大規模リファクタリングを避け、小さく可逆な変更を優先する。
12. 不明な仕様を推測だけで変更せず、現行コード、保存データ例、関連画面から確認する。

## 5. やってはいけないこと

- 明示的な依頼なしに、両アプリを同時に大きく書き換えない。
- 単一HTMLを一度にReactやTypeScriptへ全面移行しない。
- 見た目の整理を理由に、計算ロジックや保存形式まで同時に変更しない。
- `localStorage.clear()` を使わない。他アプリや同一オリジンの無関係なデータを消さない。
- 既存キーを削除してから新形式を書き込む移行を行わない。
- 保存失敗、JSON解析失敗、通信失敗を成功扱いにしない。
- ユーザー確認なしに、実データをデモデータで上書きしない。
- バックアップ形式を検証なしで信用しない。
- APIキー、アクセストークン、個人情報、位置情報をソースコードへ埋め込まない。
- APIキーや秘密情報をログ、画面、エラーメッセージ、バックアップへ不用意に含めない。
- ブラウザから第三者APIへ直接秘密鍵を送る方式を新規追加しない。
- CORSを回避するために無制限な公開プロキシを作らない。
- CDNや外部APIが常に利用可能だと仮定しない。
- iPhone Safariで未確認のAPIやCSSだけを前提に、既存フォールバックを削除しない。
- 依頼されていない整形、ファイル名変更、ディレクトリ再編、ライブラリ更新を混ぜない。

## 6. localStorage / IndexedDB / APIキー / 外部通信

### localStorage

- `localStorage` は同期APIであり、容量制限、プライベートブラウズ、OSによるデータ削除、保存失敗を考慮する。
- 書込前後で成功を確認できる設計を優先し、失敗を空の `catch` で隠さない。
- 大きな配列やストリームを単一キーへ追加する場合は、保存サイズと性能への影響を確認する。
- キー名を変更する場合は旧キーを読み取れる移行期間を設ける。
- データ削除操作は対象キーを限定し、確認ダイアログとバックアップ導線を維持する。

### IndexedDB

- IndexedDB化は将来の改善候補ですが、依頼なしに導入しない。
- 導入する場合は一括移行ではなく、リポジトリ層を設けて旧 `localStorage` と並行読込できる段階移行にする。
- 移行完了を確認するまで旧データを削除しない。
- スキーマバージョン、トランザクション失敗、部分移行、ロールバックを考慮する。
- RunOSでは活動本体とFITストリームを分け、wanoku-naviでは設定、ログ、予測、外部シグナルを分ける方針を優先する。

### APIキー

- APIキーはクライアント配布物に埋め込まない。
- ブラウザの `localStorage` に保存したAPIキーは、同一オリジン上のスクリプトから参照可能であることを前提に扱う。
- 将来的には、キーを保持するCloudflare Worker等の管理されたバックエンド経由を推奨する。
- Worker側でも許可オリジン、レート制限、入力サイズ、接続先、エラー出力を制限する。

### 外部通信

- 通信処理にはタイムアウト、HTTPステータス確認、JSON検証、失敗表示、フォールバックを設ける。
- 外部レスポンスをそのまま状態やDOMへ入れない。正規化と検証を行う。
- JMA、Open-Meteo、Anthropic、地図タイル、RSSの仕様変更を個別アダプター内へ閉じ込める方針を採る。
- `file://` とHTTPSではCORS、位置情報、保存、ダウンロードの挙動が異なる。両者を混同しない。

## 7. iPhone Safari を重視する

変更時はデスクトップChromeだけで完了と判断しないでください。少なくとも以下を考慮します。

- `viewport-fit=cover` と `safe-area-inset-*`
- 下部固定タブとホームインジケーターの重なり
- Safariのアドレスバー表示・非表示によるビューポート変化
- `100vh`、固定要素、sticky要素、ボトムシートの高さ
- ソフトウェアキーボード表示中の入力欄、スクロール、閉じる操作
- タップ領域、横スクロール、長いラベル、文字拡大
- 位置情報の権限拒否、再許可、タイムアウト
- ファイル選択、複数FIT取込、JSON読込
- Blob URLによるバックアップ保存とファイル名
- `localStorage` の保存失敗やデータ消失
- オフライン、低速回線、CDN障害、API失敗
- 大量データ処理中のフリーズ、メモリ圧迫、強制再読込
- ホーム画面追加時と通常Safariタブでの保存領域・表示差

可能なら、デスクトップのWebKit系テストに加えて実機iPhone Safariで確認します。実機確認できない場合は、その事実を変更報告に明記してください。

## 8. 変更後に必ず書くべき確認事項

作業完了時の報告には、最低限次を記載してください。

1. 変更したファイル
2. 変更の目的と要点
3. 変更していない範囲
4. 既存データ形式への影響
5. `localStorage` / IndexedDBへの影響
6. 外部通信、API、APIキーへの影響
7. iPhone Safariへの影響
8. 実施した確認またはテスト
9. 実施できなかった確認
10. 既知のリスクとロールバック方法

報告例:

```text
変更ファイル:
- runos/RunOS v100 apex.html

確認:
- 既存の meridian.v1 形式は変更なし
- デモデータ、手入力、JSONバックアップ読込を確認
- 外部通信とAPIキーの扱いに変更なし
- iPhone実機は未確認。Safari相当の画面幅で表示のみ確認

ロールバック:
- 今回の差分だけを戻せば旧動作へ復帰可能
```

## 9. 将来のReact / TypeScript化

React化やTypeScript化は将来の選択肢ですが、現時点では提案と準備に留めます。

移行する場合も、全面置換ではなく次の順を推奨します。

1. 現行計算結果のテストデータを作る。
2. 型と純粋な計算関数をUIから分離する。
3. 保存処理と外部通信をアダプター化する。
4. 既存HTMLと並行して画面単位でReactへ移す。
5. iPhone Safariで旧版と新版を比較する。
6. データ移行とロールバックが確認できてから旧実装を削除する。

「Reactへ移すこと」自体を目的にせず、既存データと計算結果を守りながら、変更を検証可能にすることを目的としてください。
