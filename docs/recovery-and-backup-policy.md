# 保存障害・破損復旧・バックアップ方針

最終確認日: 2026-07-09  
対象: RunOS / wanoku-navi

この文書は両アプリに実装済みの共通方針と相違点をまとめる。各アプリ固有の詳細は `docs/runos-storage.md` と `docs/wanoku-storage.md` を参照する。

## 1. 現行方針

両アプリはローカルファーストであり、永続保存の中心はブラウザの `localStorage` である。保存安全性について現在共通している方針は次のとおり。

1. 保存前にJSON化し、概算サイズをconsoleへ記録する。
2. JSON変換、永続書込、破損退避の失敗を無言で処理しない。
3. 保存失敗時は成功通知を出さない。
4. 破損した元文字列を別キーへ退避する。
5. 破損検知後は元キーを既定値やデモデータで自動上書きしない。
6. 画面上に障害状態とバックアップ導線を表示する。
7. 元文字列の書き出しと、現在状態の通常バックアップを別の操作として提供する。
8. 既存の正常保存キー名とJSON形式を維持する。

## 2. 実装の相違

| 項目 | RunOS | wanoku-navi |
|---|---|---|
| 保存単位 | `DB` 全体を単一キー | 用途別の複数キー |
| 主な正常キー | `meridian.v1` | `spots`、`logs`、`settings` 等 |
| 破損退避キー | `meridian.v1.corrupt.<timestamp>` | `<元キー>.corrupt.<timestamp>` |
| 破損ロック | アプリ全体の保存を停止 | 破損したキーだけ停止 |
| 失敗後のメモリ利用 | 一時 `DB` として存在 | `Store.mode === 'mem'` として明示 |
| 保存結果 | `save()` の真偽値 | `Store.set()` の真偽値 |
| 通知関数 | `notifySaveResult()` | `notifyStoreResult()` |
| 通常バックアップ | `runos-backup-YYYY-MM-DD.json` | `wanoku-v27.json` |
| 破損元書き出し | `runos-corrupt-<detectedAt>.txt` | `wanoku-corrupt-<key>-<detectedAt>.txt` |

## 3. 保存失敗時

### RunOS

`save()` はJSON変換または `localStorage.setItem()` に失敗すると `false` を返す。画面には「保存できませんでした」または具体的な保存警告を出し、設定からバックアップを書き出すよう案内する。consoleには `[RunOS storage]` の接頭辞で原因と概算サイズを出す。

RunOSは書込失敗後に永続的な「メモリモード」へ切り替える実装ではない。現在の `DB` はページ内に残るが、次の `save()` は再びlocalStorageへの保存を試みる。ただし破損リカバリ中は、明示的にロックを解除するまで試行せず `false` を返す。

### wanoku-navi

永続書込に失敗すると `Store.mode` が `mem` へ切り替わる。以後の変更はページ内メモリにだけ保持され、`Store.set()` は `false` を返す。画面には「一時保存」「このセッション限り」「再起動前にバックアップ」と表示する。consoleには `[wanoku storage]` の接頭辞で原因、キー、概算サイズ、モードを出す。

ページを閉じるか再読み込みすると、メモリだけにある変更は失われる。

## 4. 破損検知時

破損とは、JSON解析に失敗した場合、または実装済みの必須形状検証に失敗した場合を指す。

共通処理:

1. 読み取った元文字列を退避キーへ保存する。
2. 成否と退避キー名をconsoleへ出す。
3. 元の正常キーは変更しない。
4. 画面に破損検知と退避結果を表示する。
5. 元文字列を書き出すボタンを表示する。

退避キーの `<timestamp>` は `Date.now()` のミリ秒値。退避に失敗しても、現在ページのメモリ内に元文字列が保持されている間は専用ボタンから書き出しを試みられる。

## 5. リカバリロック

### RunOS: 全体ロック

`STORAGE_RECOVERY.active` の間は `meridian.v1` へのすべての `save()` を停止する。デモデータは一時表示されるが、自動保存されない。

ロックを解除して主キーを書き換える実装済みの経路は、警告内の `デモデータで再開始`。確認ダイアログへの同意後にだけ保存を試みる。失敗すればロックを復元し、退避キーは残す。

### wanoku-navi: キー単位ロック

破損したキーを `blocked` へ追加する。`Store.isBlocked(key)` が `true` の間、そのキーへの永続書込を停止する。他の正常キーは保存できる。

ロック解除を伴う実装済みの経路は、ユーザーが開始するバックアップ読込と全消去。これらは `Store.allowOverwrite()` を呼んでから複数キーを書き込む。複数キー保存はトランザクションではない。

## 6. ユーザー通知

成功通知は永続保存結果を確認した経路だけで表示する。

RunOS:

- 正常保存: 呼出元固有の成功メッセージ
- 通常失敗: `保存できませんでした`
- リカバリ中: `リカバリ中のため保存を停止しています`

wanoku-navi:

- localStorage正常保存: 呼出元固有の成功メッセージ
- `window.storage` 受付: `保存処理を受け付けました`
- メモリ保存: `一時保存です。このセッション限りの保存です`
- 破損ロック: `破損データ保護中のため保存を停止しています`

通知関数を使わない内部保存も残っているため、常駐警告とconsoleも確認手段として維持されている。

## 7. バックアップの種類

### 通常バックアップ

現在アプリが解釈できている状態をJSONとして出力する。別ブラウザへの移行や通常の復元に使う。

- RunOS: 設定 → `JSONで書き出し`
- wanoku-navi: 設定 → データ管理 → `バックアップ`

### 破損元データ

JSON解析や検証を行う前の元文字列をテキストとして出力する。手動調査や可能な範囲でのデータ救出に使う。通常バックアップとしてそのまま再取込できる保証はない。

- RunOS: 破損警告 → `破損データを書き出す`
- wanoku-navi: 破損警告 → `破損データを書き出す`

破損リカバリ中は、まず破損元データを保全し、その後で通常バックアップや再開始を判断する。

## 8. 復旧時の実務手順

### RunOS

1. 破損警告に表示された退避キー名を記録する。
2. `破損データを書き出す` で元文字列を保存する。
3. 必要なら設定から現在表示状態のJSONも書き出す。
4. 元データを残す必要がなければ、`デモデータで再開始` を選ぶ。
5. 確認ダイアログに同意する。
6. 成功表示後に再読み込みし、デモ状態が残ることを確認する。
7. 退避キーは自動削除されないため、必要に応じて開発者が別途管理する。

### wanoku-navi

1. 警告に表示された元キーと退避キーを記録する。
2. `破損データを書き出す` で対象キーの元文字列を保存する。
3. 設定の `バックアップ` で現在の利用可能な状態も保存する。
4. 信頼できるバックアップがあれば `読み込む` を実行する。
5. 全初期化する場合だけ `全消去` を実行する。
6. 操作後に対象キーを確認し、再読み込み後も状態が残ることを確認する。
7. 複数キーの一部だけ保存される可能性があるため、consoleも確認する。

## 9. 既知の共通リスク

- localStorageには容量制限があり、ブラウザや利用形態で上限が異なる。
- iPhone Safariのプライベートブラウズ、ストレージ削除、ホーム画面追加時の保存領域差の影響を受ける。
- Blobを使うバックアップ書き出しは、iPhone SafariのダウンロードUIやファイル保存先に依存する。
- 退避キーの一覧、世代比較、削除、選択復元を行う管理UIはない。
- 退避キー作成と主キー更新を一つのトランザクションにはできない。
- タイムスタンプはミリ秒であり、退避キーの衝突を明示検査していない。
- 通常バックアップの読込検証は、両アプリとも完全なスキーマ検証ではない。
- メモリ上の状態が更新されても永続保存されていない場合がある。
- 実機iPhone Safariを含む自動復旧テストは存在しない。

アプリ固有の追加リスクは各保存仕様書を参照する。

## 10. 変更時の確認事項

保存関連を変更した場合は最低限、次を確認する。

- 既存キー名と正常JSON形式を変更していないか
- 正常保存時だけ成功通知が出るか
- 書込例外が画面とconsoleの両方で確認できるか
- JSON破損時に元キーが自動上書きされないか
- 退避キー名と元文字列が正しいか
- 退避自体が失敗しても元キーを上書きしないか
- リカバリ中または破損ロック中の保存結果が `false` になるか
- 通常バックアップと破損元書き出しの両導線が使えるか
- 再読み込み後に永続状態が期待どおりか
- iPhone Safariで警告、ボタン、ダウンロードが操作可能か

## 11. 将来候補: IndexedDB

IndexedDBは両アプリとも未実装であり、ここまでの現行方針には含まれない。

将来候補として検討する場合の原則:

- 現行localStorageを直ちに削除しない。
- 保存層をアダプター化し、旧形式を並行読込できる期間を設ける。
- 移行完了を検証してから旧キーの扱いを決める。
- スキーマバージョン、トランザクション失敗、部分移行、ロールバックを設計する。
- RunOSは大容量FITストリームとメタデータの分離を検討する。
- wanoku-naviは複数キーの論理グループをトランザクション単位へ対応付ける。

これらは設計候補であり、現時点で実装済みの機能ではない。
